Web delivery — the freshness contract

A skill + a library so every kolu-shaped web/PWA app (kolu, drishti, the next one) ships the deployed client to returning users by construction — and makes version drift visible and one-tap recoverable. Status: seedling · proposed in #1149, to build in a follow-up PR.

A continuously-deployed web app must guarantee that a returning client converges to the current build — and make any divergence visible and one-tap recoverable. Everything below is in service of that one property.

Why this exists

kolu has now attacked this same property four times — #696 (stop serving stale assets), #1125 (race-free SW reload), #1135 (the ≠ srv badge), and #1149 (the no-store shell + mobile indicator — and ultimately removing the service worker outright, which turned out to be the actual recurring cause). Each time it was re-derived from scratch, slightly differently, and a gap remained. The deepest trap (the full saga is in cache-bug.md): a service worker is a separate interception layer that the HTTP-cache directives (no-store) can't touch — it served an old build on every normal reload — and it survived on a plain-HTTP origin only because a Chrome "treat-insecure-as-secure" flag made that origin a secure context, which a protocol === "https:" check then misread. The cost each time was re-learning the contract from zero.

The lesson is the same one electricity.html already records: conventions-as-prose drift; infrastructure that graduates into a shared package does not. @kolu/surface proved it — it's the reactive transport of drishti. Client-delivery freshness is the next electricity: domain-agnostic, hides a genuinely hard volatility (browser/SW/proxy caching across deploys), and graduates so the next app plugs in instead of re-deriving. → add to the tracker in electricity.html once built.

The invariants it encodes

In priority order. #1 is load-bearing — get the freshness contract right and the rest are graceful degradation, not the difference between working and broken.

1
One mutable entry point; everything else immutable. The shell (index.html) is the only never-cached resource (Cache-Control: no-store); every other asset is content-hashed and immutable; a missing /assets/* hash 404s rather than falling through to the HTML shell. Get the layering right and staleness is structurally impossible — no SW, proxy, or browser quirk can serve an old app, because the one document that names the bundle is always re-fetched.
2
Build identity is first-class and single-sourced. Client and server stamp the same commit from one place; the server exposes it (server.info). You can't detect skew you don't measure.
3
Skew is visible and recoverable, on every form factor. When client ≠ server, show a durable indicator (not a transient one a backgrounded tab misses) and offer a reload that actually lands fresh — desktop rail and mobile.
4
A service worker is an opt-in you own end-to-end — including retirement — or you ship none. A SW intercepts navigations in front of the network, so the HTTP-cache contract (#1) can't reach it: a stale worker serves an old build on a normal reload regardless of no-store. If you register one you own its whole lifecycle — the update race (prompt → skipWaiting → controllerchange) AND a way to unregister a deployed worker (a self-destructing sw.js the browser's own update check installs, plus an on-load getRegistrations().unregister()); gating new registration never removes an existing one. Gate on window.isSecureContext, never location.protocol === "https:" (that misses http://localhost and flag-secured origins — the mistake that orphaned kolu's worker). For an always-connected app a SW earns nothing (no offline; the immutable HTTP cache already covers speed), so kolu ships none.
5
The delivery layer is observable, and the bug lives in the browser. The triage shortcut: normal reload stale, hard reload fresh → a cached shell or a service worker. Distinguish them in the browser, not by reasoning about the origin: the Network panel's Size column reads (ServiceWorker) when a worker is serving, and navigator.serviceWorker.getRegistrations() lists it. Never assume "plain HTTP ⇒ no SW" — a Chrome insecure-origin flag (or localhost) makes the API available. The decisive facts are browser-side; server-side curl can show the server is perfect while the browser still serves stale.

Two artifacts

The judgment lives in a skill; the implementation lives in a library, so neither rots. The skill says "reach for the library, and here's why each piece exists"; the library keeps the correct code from being re-derived wrong.

① The library — the correct code, importable

Mirror @kolu/surface's shape: one package, two sub-path entrypoints.

// server: Hono middleware — the cache-control policy + SPA fallback (invariants 1, 2)
import { freshnessRoutes } from "@kolu/web-delivery/server";
freshnessRoutes(app, { clientDist, commit });   // no-store shell · immutable assets · 404 misses · /server.info

// client (SolidJS): skew detection + reload UX (invariant 3) + SW retirement (4)
import { retireServiceWorker, clientStale, StaleBadge, ReloadPrompt } from "@kolu/web-delivery/solid";
retireServiceWorker();   // unregister any worker an earlier build left; pairs with the package's self-destructing sw.js

Each app supplies its build commit (one stamp, used both sides) and renders <StaleBadge/> / <ReloadPrompt/> wherever its chrome lives. The package also ships the self-destructing sw.js (served at /sw.js, no-cache) that retires a legacy worker via the browser's own update check. #1149 already factored the kolu pieces toward this seam — cacheControl.ts (the directive map), pwa.ts (retireServiceWorker + the no-SW freshness signals), public/sw.js, and the shared ui/StaleBadge — so extraction is a lift, not a rewrite.

② The skill — the judgment + review lens, distributed via apm

An author guide (scaffolding a new app's delivery layer → reach for the library, set the SW policy deliberately) and a review lens (auditing an existing app → check the five invariants, run the normal-vs-hard-reload triage). Distributed through the apm registry so kolu / drishti / the next app all pull the same one. Seed it with #1149 as the worked example.

Names — recommendation

The unifying concept is "the client the user runs is the one we deployed." That's delivery freshness, broader than caching alone (it also covers version stamping + skew UX + the SW decision), so the name shouldn't say "cache".

ThingRecommendedAlternativesWhy
Library@kolu/web-delivery@kolu/freshness · @kolu/deploy-syncDescriptive, domain-agnostic, fits the @kolu/* electricity family (surface, log, html-escape). Covers all four moving parts, not just the cache.
Skill/web-delivery/ship-fresh · /freshnessShare the library's name for discoverability — the skill's first instruction is "use @kolu/web-delivery."

If the server-serving and the client-skew-UX feel like two volatilities later, the library can split (/server stays cache+serving; a @kolu/solid-version-skew peels off the indicator). Start as one package with two entrypoints; let usage tell us.

Phasing (the follow-up PR)

  1. Extract the library from the kolu code #1149 already shaped — server cache-control middleware + SPA fallback, version stamping, clientStale/StaleBadge/reload prompt, retireServiceWorker + the self-destructing sw.js — into @kolu/web-delivery/{server,solid} (plus the static sw.js), and make kolu consume it (proving the seam, like surface).
  2. Write the skill (author + review lens), distribute via apm, seed with #1149.
  3. Graduate to drishti — port it onto the library; that's the acceptance test that the package is genuinely app-agnostic (the bar from electricity.html).
  4. Register it in electricity.html as a shipped electricity.

The one-line answer to "how do we do the right thing in all of them": stop re-deriving the freshness contract per app. Encode invariant #1 (one never-cached shell, everything else immutable) as importable code, wrap the judgment in a distributable skill, and let drishti import instead of re-learn.
