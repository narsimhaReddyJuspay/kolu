Code browser preview — formats today, and the ones to add

The Code tab can open any file in the repo, but "open" means two different things: show its source (syntax-highlighted text) or show its rendered form (the image, the page, the document). Markdown sat in the gap — a README.md used to open as syntax-highlighted source (the screenshot that prompted this plan), never as a rendered document. The organizing idea of this plan is a Source ⇄ Rendered toggle: for any file that has both a source and a rendered form, the user picks. Markdown is now the first lit toggle (#1093, shipped) — it defaults to Rendered, one click to source; HTML and SVG join it next.

The one-line summary. Previewing is decided at a single seam — a file is classified, the fsReadFile wire carries a kind discriminator, and a dispatcher routes kind to a presenter. The toggle is available exactly when a file has both a source and a rendered form. Markdown is the easy case (already text on the wire). HTML/SVG are the instructive case (text-backed but delivered as binary/URL today).
The thesis — invent electricity, don't wire the house. Lowy's electricity analogy: a home shouldn't generate its own power per appliance; a utility encapsulates the volatility of generation behind a stable interface (the outlet), and appliances just plug in. Today Kolu is pre-electricity — each preview format would wire its own rendering into the right-panel. This plan instead extracts the preview capability into standalone, Kolu-agnostic packages — a generic file-viewer "grid" and pluggable renderer "appliances" — and leaves Kolu plugging into the outlet. The payoff is counterintuitive: even though we add a feature (the toggle + Markdown + HTML/SVG source), the Kolu app code gets smaller, because the render mechanics that today live tangled in right-panel/ and intent/ move out to reusable libraries beside @kolu/solid-pierre. The hidden insight: kind conflates two independent questions

The text/binary discriminator answers "how do I fetch and render this by default." But the real structure underneath is two orthogonal yes/no axes — and the Source ⇄ Rendered toggle is meaningful exactly at their intersection.

The feature — a Source ⇄ Rendered toggle, Markdown first

The toggle is the real deliverable; Markdown is the first format to wear it because it's the highest-value, lowest-cost case and rehearses the control the others (HTML, SVG, later CSV) reuse.

Markdown needs no new wire kind confirmed by P3

Markdown is text — it already arrives as {kind: "text", content, truncated}. The render decision is purely client-side and reversible: a .md file should default to rendered, but the user must be able to flip to source (to see exact syntax, copy raw, or select lines for comments). So Markdown is just a renderer appliance plugged into the grid (next section), entirely above the wire. P3 confirmed this in the build: no new kind, no defaultMode — Markdown stayed kind:"text" and the dispatcher passed a one-entry rendered={[markdownRenderer]} matched by isMarkdown; FileView's existing "both forms → default Rendered, show the toggle" rule did the rest, and non-markdown text matched nothing so it stayed source-only.

3a · Comments on rendered Markdown — the way through

v1 shipped comments only in source view: CommentTextSurface wraps Pierre's CodeView and gives char-range selection there, while the rendered document is read-only (a new e2e scenario documents that you flip to source to comment). The original plan flagged the real problem — anchoring a comment to a character range in rendered output. Here is the concrete path through it, and it costs no new model.

Kolu already has one anchoring model used in two surfaces: the W3C TextQuoteSelector ({quote, prefix, suffix}) and the pure functions extractQuote(range, root) / findQuote(text, locator) in @kolu/artifact-sdk/core. The source comment surface uses it; the in-iframe HTML annotator uses it (re-exported via the SDK's client entry). Rendered Markdown is the third surface — and the simplest of the three, because:

  • Markdown renders inline in the parent document, not in a sandboxed iframe. So there is no postMessage bridge — unlike the HTML-artifact path, which must marshal the selection across an opaque-origin frame. A plain DOM Range over the rendered Markdown nodes feeds extractQuote directly; findQuote re-locates it against the node text on rehydrate.
  • The locator is durable across re-renders by construction (it's text + context windows, not a DOM path), which is exactly the property a rendered tree needs.

Volatility split (Lowy): the generic mechanism — turn a user selection over a rendered subtree into a locator, and paint a locator back as a highlight overlay — is Kolu-free and belongs as a rendered-annotation hook in @kolu/solid-fileview (any consumer annotating a rendered document wants it). The comment feature — the tray, threads, persistence — is Kolu's volatility and stays in Kolu's renderer wiring, the same place the iframe comment bridge lives today. Net: a small hook in the library, a tray adapter in Kolu, and one anchoring model shared across all three surfaces instead of a fourth invented for Markdown.

Invent the grid — extract three packages, slim Kolu

The naïve build wires each format into Kolu's right-panel/: a BrowseMarkdownView, then a BrowseHtmlView, each a Switch arm. That's the pre-electricity house — generation tangled into the wiring. Two things vary independently and must be encapsulated apart (Lowy), and neither is about Kolu:

  • How a document renders (marked → safe Solid nodes) — a self-contained capability, volatile in its own right (the markdown spec, sanitization).
  • The source ⇄ rendered viewer (the toggle, which modes are available, which renderer matches) — a generic mechanism that any file viewer needs, with zero Kolu knowledge.

So the build is: extract these as standalone packages beside @kolu/solid-pierre, and let Kolu's right-panel become a thin adapter that plugs the fsReadFile wire into them. That is the electricity: a stable outlet, appliances plugged in, the house rewired smaller.

Decoupling the source renderer too (Lowy, all the way down). FileView does not hard-depend on a syntax highlighter — the source view is an injected renderer like any other. Kolu passes one backed by @kolu/solid-pierre (already a package) carrying Kolu's theme. So @kolu/solid-fileview has no rendering dependencies at all: it is pure mechanism (toggle + registry), and every concrete renderer — source included — is an appliance the consumer plugs in. A non-Kolu app could plug a CodeMirror source renderer and a KaTeX appliance into the same outlet.

What stays in Kolu (its own volatility, kept thin)

The boundary is the outlet's data shape. Kolu owns only what is genuinely Kolu:

  • The wire adapter — map fsReadFile's {kind, content?, url?} to FileView props. This is the only file-preview code left in right-panel/, and it's a projection, not logic.
  • The renderer set Kolu injects — pierre-source (themed), markdown, image, and an iframe renderer wrapped with the artifact-sdk comment bridge. Comments are a Kolu feature, so they live in Kolu's renderer construction, not in the library.
  • Server-side classification — which paths get text vs a URL on the wire. A fetch concern; lives in a preview module (not kolu-git).

Net: Kolu deletes the per-strategy presenters and the markdown core, and adds a thin adapter + a renderer-list literal. The right-panel preview surface and the intent markdown surface both shrink, while gaining the toggle.

HTML / SVG need the wire to carry both source and URL

HTML and SVG are text on disk, but the Code tab renders them in an allow-scripts opaque-origin iframe pointed at a server-built URL — and for HTML the server splices the artifact-sdk comment bridge into the response at that route. So the render path genuinely needs the URL; it can't srcdoc the raw bytes without losing the bridge and the security envelope. To also show source, the client needs the text the iframe path never carried.

The clean move is a third fsReadFile variant for the toggle set — text-backed and iframe-rendered:

{ kind: "renderable", content, truncated, url }
//        ^source view ──────────┘          └── iframe render

Markdown stays kind:"text" (no URL — it renders client-side from content); images/PDF stay kind:"binary" (no content — there's no source). The discriminator now reads off the same two axes the taxonomy named: text = source-only, binary = rendered-only, renderable = both. This keeps the "every variant is exactly one cell of the grid" property the current partition already has.

The accounting: we add two packages and a feature, yet the Kolu app tree (client/, server/, kolu-git) ends up with fewer lines and one fewer responsibility — exactly the electricity payoff. The complexity didn't vanish; it moved behind an outlet and stopped being Kolu's to maintain.

Bottom line: the feature is a Source ⇄ Rendered toggle, offered wherever a file has both forms. Markdown is now the first lit toggle (#1093) — and lighting it cost a single library appliance plus a one-entry renderer list, because the grid (@kolu/solid-markdown + @kolu/solid-fileview, beside @kolu/solid-pierre) was already built and the "both forms → default Rendered" rule already lived in the outlet. HTML/SVG are next, then the markdown appliance grows (comments on the rendered doc, Shiki fences, anchors, math/diagrams) — every one an appliance on the grid, never a rewiring of the house. Adding a format keeps making Kolu's own code simpler, not larger.
