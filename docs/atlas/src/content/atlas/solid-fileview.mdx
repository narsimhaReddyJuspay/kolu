The Code tab can *open* any file, but "open" means two things: show its
**source** (syntax-highlighted text) or its **rendered form** (the image, the
page, the document). Markdown sat in the gap ‚Äî a `README.md` opened as source,
never as a rendered document. The organizing idea is a **Source ‚áÑ Rendered
toggle**: for any file with *both* forms, the user picks. Markdown is the first
lit toggle (<PrLink pr={1093} />, shipped); HTML and SVG join next.

<Callout kind="note" title="The one-line summary">
Previewing is decided at a single seam ‚Äî a file is classified, the `fsReadFile`
wire carries a `kind` discriminator, and a dispatcher routes `kind` to a
presenter. The toggle is available exactly when a file has *both* a source and a
rendered form. Markdown is the easy case (already `text` on the wire); HTML/SVG
are the instructive case (text-backed but delivered as `binary`/URL today).
</Callout>

<Callout kind="accent" title="The thesis ‚Äî invent electricity, don't wire the house">
Lowy's electricity analogy: a home shouldn't generate its own power per appliance;
a utility encapsulates the volatility of *generation* behind a stable interface
(the outlet), and appliances just plug in. Today kolu is pre-electricity ‚Äî each
preview format would wire its own rendering into the right-panel. This plan
**extracts the preview capability into standalone, kolu-agnostic packages** ‚Äî a
generic file-viewer "grid" and pluggable renderer "appliances" ‚Äî and leaves kolu
plugging into the outlet. The payoff is counterintuitive: even though we *add* a
feature (the toggle + Markdown + HTML/SVG source), **the kolu app code gets
smaller**, because the render mechanics move out to reusable libraries beside
`@kolu/solid-pierre`.
</Callout>

## What can be previewed today

Three render strategies, chosen below the `fsReadFile` wire boundary by file
extension. The classifier is deliberately node-free so server and client import
the *same* lists.

| Strategy | Wire `kind` | Renderer | Extensions |
| --- | --- | --- | --- |
| **Source text** (default) | `text` | `BrowseFileView` ‚Üí Pierre `CodeView` (Shiki), wrapped in `CommentTextSurface` for line/range comments. | *Everything not listed below.* Truncated past 1 MB. |
| **Raster image** | `binary` | `BrowsePreviewView` ‚Üí plain `<img>` on a checkerboard. No iframe ‚Äî image bytes can't execute. | `.png .jpg .jpeg .gif .webp .ico` |
| **Sandboxed document** | `binary` | `BrowsePreviewView` ‚Üí `allow-scripts`, opaque-origin `<iframe>`. HTML gets the artifact-sdk comment bridge; SVG/PDF verbatim. | `.html .htm .svg .pdf` |

The two binary sets are disjoint; their union is `BINARY_PREVIEWABLE_EXTENSIONS` ‚Äî
the partition is structural, so a new previewable format lands in exactly one
category.

### The hidden insight: `kind` conflates two independent questions

The `text`/`binary` discriminator answers "how do I fetch and render this by
default." The *real* structure is two orthogonal yes/no axes ‚Äî and the toggle is
meaningful exactly at their intersection.

| Format | Has a source? | Has a rendered form? | What the user gets |
| --- | --- | --- | --- |
| plain code `.ts .rs .py` | ‚úì | ‚úó | Source only (today's default ‚Äî correct) |
| **Markdown** `.md` | ‚úì | ‚úì (client md render) | **Toggle** ‚Äî default Rendered |
| **HTML** `.html .htm` | ‚úì | ‚úì (iframe) | **Toggle** ‚Äî default Rendered |
| **SVG** `.svg` | ‚úì | ‚úì (iframe) | **Toggle** ‚Äî default Rendered |
| images `.png .jpg` | ‚úó | ‚úì (`<img>`) | Rendered only ‚Äî *no source exists* |
| PDF `.pdf` | ‚úó | ‚úì (iframe) | Rendered only ‚Äî *no source exists* |

The toggle is offered **iff both columns are ‚úì**. This is a property of the
format, not a per-presenter convention.

## The feature ‚Äî a Source ‚áÑ Rendered toggle, Markdown first

### Markdown needs no new wire `kind` <Pill variant="ok">confirmed by P3</Pill>

Markdown *is* text ‚Äî it arrives as `{kind: "text", content, truncated}`. The
render decision is purely client-side and reversible: default **rendered**, one
flip to **source**. So Markdown is a renderer appliance plugged into the grid,
above the wire. *P3 confirmed it:* no new `kind`, no `defaultMode` ‚Äî the
dispatcher passed a one-entry `rendered={[markdownRenderer]}` matched by
`isMarkdown`; `FileView`'s "both forms ‚Üí default Rendered, show the toggle" rule
did the rest.

### 3a ¬∑ Comments on rendered Markdown ‚Äî the way through

v1 shipped comments only in **source** view (`CommentTextSurface` wraps Pierre's
`CodeView`); the rendered document is read-only. kolu already has *one* anchoring
model in two surfaces: the W3C **`TextQuoteSelector`** (`{quote, prefix, suffix}`)
+ `extractQuote`/`findQuote` in `@kolu/artifact-sdk/core`. Rendered Markdown is the
third surface ‚Äî and the *simplest*, because:

- Markdown renders **inline in the parent document**, not in a sandboxed iframe ‚Äî so there's **no postMessage bridge**. A plain DOM `Range` feeds `extractQuote` directly; `findQuote` re-locates on rehydrate.
- The locator is durable across re-renders by construction (text + context, not a DOM path).

Volatility split (Lowy): the *generic mechanism* (selection ‚Üí locator, locator ‚Üí
highlight) is a **rendered-annotation hook in `@kolu/solid-fileview`**; the
*comment feature* (tray, threads, persistence) stays kolu's. One anchoring model
across all three surfaces, not a fourth invented for Markdown.

### 3c ¬∑ More Markdown features ‚Äî appliances on the grid

Each lives *inside the appliance* (`@kolu/solid-markdown`); the host never changes.
Grounded against what `marked` ships (`{gfm: true, breaks: true}`):

| Feature | Status today | The work |
| --- | --- | --- |
| GFM tables / task lists | `marked` parses; renderer emits `<table>` + checkbox nodes. | Styling polish for `document`; verify checkboxes read as decorative. |
| Strikethrough / autolinks | Parsed by GFM. | Confirm a `del` case + bare URLs through `safeHref`; style if missing. |
| Syntax-highlighted code fences | Plain `<pre><code>`. | **3b** ‚Äî Shiki on the `code` token; highlighter in-repo via `@kolu/solid-pierre`. |
| Heading anchor links | No ids. | Slug-id each heading + clickable `#` permalink. |
| Relative *file* links | `safeHref` allows; clicking does nothing. | Resolve a relative path to *opening that file in the Code tab* (kolu link policy, injected). |
| Frontmatter | Renders as a literal block. | Strip-and-ignore by default. |
| KaTeX math ¬∑ Mermaid | Not handled. | *Optional appliances* plugged into the same outlet later ‚Äî no host changes. Demand-gated. |

## Invent the grid ‚Äî extract three packages, slim kolu

The na√Øve build wires each format into kolu's `right-panel/` as a `Switch` arm ‚Äî
the pre-electricity house. Two things vary independently and must be encapsulated
apart (Lowy), and neither is *about kolu*:

- **How a document renders** (marked ‚Üí safe Solid nodes) ‚Äî volatile in its own right (the spec, sanitization).
- **The source ‚áÑ rendered viewer** (the toggle, mode-availability, renderer match) ‚Äî a generic mechanism any file viewer needs, with *zero* kolu knowledge.


### The three packages (Lowy: each encapsulates one volatility)

| Package | Encapsulates | The outlet | What it obviates in kolu |
| --- | --- | --- | --- |
| `@kolu/solid-markdown` <Pill variant="todo">new</Pill> | marked + GFM + `safeHref` + md‚ÜíSolid styling. Volatile: the spec, the sanitizer. | `<Markdown markdown variant="inline"\|"document" />` | The ~250 LOC token-walk core inside `intent/IntentMarkdown.tsx`. |
| `@kolu/solid-fileview` <Pill variant="todo">new</Pill> | The toggle, mode-availability, renderer-registry pick. Knows nothing of oRPC, git, comments. | `<FileView path source? renderedUrl? source={Renderer} rendered={Renderer[]} />` | The render *mechanics* of `BrowseFileView` + `BrowsePreviewView` + `BrowseFileDispatcher`. |
| `@kolu/solid-fileview/renderers/{markdown,image,iframe}` | Each strategy as an independent appliance. Generic, kolu-free. | Each is a `RenderedRenderer` value. | The strategy code hand-written in `BrowsePreviewView`. |

**Decoupling the source renderer too.** `FileView` does *not* hard-depend on a
highlighter ‚Äî the source view is an injected renderer like any other. kolu passes
one backed by `@kolu/solid-pierre` carrying kolu's theme. So `@kolu/solid-fileview`
has no rendering deps at all: pure mechanism, every concrete renderer an appliance.

### HTML / SVG need the wire to carry both source and URL

HTML/SVG are text on disk but render in an `allow-scripts` opaque-origin iframe at
a server-built URL (and HTML splices the comment bridge at that route) ‚Äî the
render path genuinely needs the URL. To *also* show source, the client needs the
text the iframe path never carried. A third `fsReadFile` variant:

```ts
{ kind: "renderable", content, truncated, url }
//        ^source view ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò          ‚îî‚îÄ‚îÄ iframe render
```

Markdown stays `kind:"text"` (no URL); images/PDF stay `kind:"binary"` (no
content). The discriminator now reads off the same two axes: `text` =
source-only, `binary` = rendered-only, `renderable` = both.
