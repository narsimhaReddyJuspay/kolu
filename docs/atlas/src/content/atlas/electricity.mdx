<Callout kind="good" title="What counts as electricity">
Löwy's receptacle — hide the volatility behind a socket you plug into. Three tests at once: ① **domain-agnostic** — no terminal/canvas/git/xterm leaks into the interface; ② **hides a hard volatility** — transport, reconnect, GPU-context loss, persistence — not a 3-line `===`; ③ **graduates** — ships as `@kolu/*` and a *different* app plugs in. The proof of ③ is real, not aspirational. [`@kolu/surface`](https://kolu.dev/blog/surface-framework/) is **kolu's own** client↔server transport first — and it has since picked up a **second consumer** in [srid/drishti](https://github.com/srid/drishti) (a fleet process-monitor; browser ↔ Bun ↔ ssh-agent) that vendors it from this repo and declares its own per-host schema with none of kolu's domain. The second consumer isn't the *point* of the extraction — it's the *proof* it was real electricity: surface never noticed the domain changed.
</Callout>

<Callout kind="danger" title="How a session mis-identifies electricity — three traps, learned the hard way">
The live example: asked to add back/forward to the Code-tab preview, a session proposed extracting a **history stack**. Wrong three ways, each caught only by re-asking the test above — and the [solid-browser](solid-browser) plan is the corrected end-to-end.

- **Naming an organ, not the concept.** A history stack is a *transformer* — a working part *inside* a larger thing. The electricity was the **browser** (location + history + link-nav + render-host); history is one chamber of it. Test: *is the name self-describing standalone?* "history" → history **of what?**; "browser" → obvious. If you must append "…of X" to explain the name, you've named a part, not the receptacle.
- **A module in `client/src/` is not electricity.** Location *is* structure (Hickey): a generic-looking helper living inside the app is **part of the app**, however agnostic the code reads. Electricity is its **own package**, the dependency arrow pointing *out* — `client → @kolu/x`, never back. (`createSharedRoot` sits in `client/src` marked "publish?" — a leaf *awaiting* graduation, **not** a precedent for calling in-app modules electricity.)
- **Agnostic ≠ electricity — apply test ② honestly.** A clean, reusable, agnostic primitive that hides only a *bounded algorithm* (a stack, a debounce, an `===`) is a **leaf** (`nonempty`/`html-escape`-tier), not a receptacle. "I found a tidy generic module" is the feeling that *precedes* the mistake. Ask what **hard** volatility it hides — transport, GPU-context loss, sandboxing, reconnect — before reaching for `@kolu/`.
</Callout>

> "Decompose based on volatility." Functional decomposition splits a system by what it *does*; volatility-based decomposition isolates each area likely to *change* behind a stable interface — the way a body keeps blood-pressure, pulse, and salinity volatility "encapsulated behind the service called the heart." An *electricity*, here, is one such volatility pulled out into its own package. — Juval Löwy, [Volatility-Based Decomposition](https://www.informit.com/articles/article.aspx?p=2995357&seqNum=2) (excerpt from *Righting Software*).

<Callout kind="warn" title="Re-evaluating @kolu/artifact-sdk — its electricity framing shifted (open; resolve outside this PR)">
"Anchoring" (the original <PrLink pr={922} /> framing) now under-describes the
package. It holds **two** volatilities: (1) the **anchoring core** (`core/` — W3C
quote extract/re-find/highlight; pure and agnostic, speaks only
`Range`/`Document`/`ShadowRoot`), and (2) the **sandboxed-iframe parent↔frame
bridge** (`client/bridge.ts` + `iframe/` — opaque-origin postMessage with
`event.source` identity as the only trust boundary). The bridge is the *harder*
volatility, and it has quietly accreted clients: comments (selection →
`SelectMsg` → highlights), then **in-frame navigation** (`observeIframeNavigation`),
then **back/forward input** (`observeIframeHistory`, <PrLink pr={1191} />). The
lens-debate in <PrLink pr={1191} /> already reframed the package's *stated*
identity to "the in-iframe sandbox bridge, with comments as one client" — but
that was a docs fix, not a decomposition.

**The open question (work it outside this PR):** is "talk to code inside an
opaque-origin sandboxed frame" one sound receptacle — a transport-grade
volatility with comments / navigation / input as interchangeable clients — or a
topic-bundle that should split the agnostic **anchoring core** from the **bridge
transport**? Note the smell that prompted this: the back/forward client landed in
this package only by *physical necessity* (the one script inside the sandbox is
this one), which is exactly the "easy to put it here" the three traps above warn
against. Flagged, not decided.
</Callout>

Plus external electricity correctly leaned on instead of hand-rolled: **Corvu** (dialog/drawer/focus-trap/scroll-lock), **@solid-primitives/\*** (resize-observer, media). Origin: a 41-agent Hickey/Lowy audit (16 candidates) → its "ship now" tier landed in <PrLink pr={1089} />; a framework-scale re-test + adversarial skeptic surfaced the missed `solid-xterm`.

## Next: extract @kolu/solid-xterm
